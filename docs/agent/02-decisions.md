# 决策记录

更新时间：2026-05-06

本文记录影响项目维护方式的关键决策。正式功能事实仍以 `docs/guide/` 为准。

## D-001 文档按 guide / agent 分区

决策：

- `docs/guide/` 放给人看的正式项目文档。
- `docs/agent/` 放给 AI 和开发连续性看的维护上下文。

原因：

- 正式文档需要稳定、可读、面向使用和维护。
- AI 需要知道当前状态、未完成任务、设计决策和交接信息。
- 两类信息混在一起会让 README 和项目文档变得像开发日志。

## D-002 根 README 是唯一主入口

决策：

- 根目录 `README.md` 保留为项目主入口。
- 不再保留 docs 根目录索引文件作为第三个入口。

原因：

- 根 README 负责介绍项目和给出最短启动路径。
- `docs/guide/00-overview.md` 负责正式文档阅读顺序。
- `docs/agent/04-handoff.md` 负责 AI / 维护交接。

## D-003 项目按完整 workspace 维护

决策：

- 项目名保持 `teleop_control`。
- 文档按完整 workspace 说明项目，不只围绕 `teleop_control_py` 单包。

原因：

- 日常运行依赖 UR driver、夹爪包、模型仓库、脚本、数据和设备规则。
- 只写主控包会让维护者低估改动影响范围。

## D-004 TODO 使用人工排序

决策：

- `docs/agent/01-todo.md` 使用 `Order` 字段控制任务顺序。

原因：

- 人工维护项目时经常需要先做某个前置任务。
- 仅靠优先级无法表达执行顺序。
